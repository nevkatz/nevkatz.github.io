Working with a large codebase

Oct 8, 2019

Intro

In recent weeks I have stepped away from SVG animation in order to focus on getting cordova mobile apps ready for rollout. The app codebase, which is built in Javascript, is quite large. Over time, I have developed some strategies for keeping the large codebase manageable and flexible. Below are a few tips:

Modules and Function Naming

Keep the code modular. Break the code into different modules, each comprising one or more files, and give each module a specific purpose. In the early days of the codebase, I started with one main module — but as the number of modules proliferated, the code in that first modules ended up getting dispersed into more specific modules.

I recommend giving each module a name that describes that purpose. And if two modules are in a similar realm but have distinct purposes, I often start them with the same word (ex. user_save and user_retrieve) so they show up together when my files are sorted alphabetically.

Name functions according to the modules they are in. I have over 100 modules in the app, and each module starts with something like current_activity. When I look at a function, I like to see what module it is a part of, so I recommend naming all functions so that they start with the name of their module. (ex. current_activity_setup.)

Comments and Variable Names

Comment where necessary but be selective. As long sa the volume of comments is something you can maintain, they are helpful, but older comments can become outdated. I recommend placing comments at the top of a given file and above your functions.

Make variable names descriptive. Make sure your variables are recognizable. Think about what you would recognize if you came back to the code after a long break. When returning to a long block of code, would you rather come back to let x = 5 or let starting_accounts = 5? Good variable names can eliminate the need for commenting, which can often become outdated.

Function Trees

Map out function trees for a given module. To keep a bird’s eye view of how the different functions relate in a module, I create a README.txt file for each module. Git

Use Git to your advantage. Especially with large codebases I like to do the following with Git:

  • check commit history with git log often to get re-oriented, especially after the weekend
  • use git branch to separate out stable points in the build
  • git tag important releases or milestones with version number and description
  • make detailed git commit messages when needed
  • using git diff when making a detailed commit to see what has changed

If I am making a short commit, I will just use git commit -m <message>. If I make a lot of changes in a lot of places, I will type git commit, which takes me into the command-line text editor. I then use git diff to view all my new changes. While going through my diff, I write a detailed commit message and make a detailed list of all my key changes.

I also try to git tag each version of the codebase to keep track of what versions I am at, and I while I work mainly on a devel branch, have a few git branches that I know are at a stable point in development.

Takeaway

Large codebases don’t have to feel like a jenga tower that is about to topple over. If you keep your code modular and are clear about where your functions are, what your variables mean, and how your functions and modules connect with each other, you can have a codebase that you enjoy coming back to and working on.
