My journey with Jekyll

Aug 10, 2019

Getting Jekyll working again with the latest version of Ruby

This site is published with Jekyll, a static site generator that publishes raw HTML files using markdown files and templates. I wanted a solution that allowed me to write and publish without getting bogged down in security updates and other types of back-end maintenance, but also allowed me to innovate on the front-end when I wanted. I was drawn to Jekyll because of how it enabled me to build a site efficiently.

Before firing up this site, I had briefly experimented with Jekyll in 2017 after hearing about it on Tom Johnson’s I’d Rather Be Writing site. and then I started it up again this month in 2019 after hearing about Becca Williams, another developer, having success with it.

Jekyll Setup

As with other tools I have run from the command line, the setup process had the usual trickiness with versions and tool chains — but after getting things up and running, I have to say that love the easy deployment process. Below I’ll detail what it was like.

The first challenge was making sure I had the latest version of Jekyll running, as I had installed Jekyll 3.6.2 two years ago to just try it out. When trying to install 3.8.6 with my Mac’s installed version of Ruby, I ran into permissions issues. So I had to download a version of Ruby that was independent of my Mac version. This proved to be the breakthrough I needed, and I was able to download and run Jekyll 3.8.6.

Running the local server

One thing that I found myself adjusting to was the static site generator. It is easy enough to use and jekyll serve starts a server process on your machine, running a local dev site in your browser. A _sites directory is generated containing your output files. One thing to note, though, is that he _sites directory does not load in the CSS if you open it without having jekyll serve running. You need that server process for it to look right.

Adding CSS

One thing that initially perplexed me was where to put the styles. Jekyll uses sass and when you run jekyll serve the sass automaticallly compiles - but it was not obvious how to add other styles initially.

I had seen a _sass directory initially in Jekyll 3.x(?) but when I updated to Jekyll 3.8.6, that the _sass directory was no longer there, nor were other key directories. I read that these were now included in the gem theme, so I used bundle show minima to find its location. I then copied _sass, _layouts, _includes, and assets over from the theme and overrode parts of the templates and styles that I wanted to change.

After exploring the directory, I decided on two approaches for styles:

  • a custom.scss file that is imported so the css shows up in main.css
  • a separate custom.css file

Wiring Git to Github Pages

I was originally storing my github-pages publishable stuff in a completely different repo and actually copying all the stuff from the _sites directory over to that repo every time I wanted to publish and push. Needless to say, I wanted omsething more efficient.

After reading about other approaches, my approach was to simply wire the _sites directory to Github pages, but because my overall jekyll directory was under site version control, I was worried about having a repo within a repo. Fortunately, .gitignore in my jekyll repo had my back, because it automatically ignores the _sites folder. So I moved the .git directory from github-pages into my _sites folder and voila — the _sites folder was under version control and wired to Git, and I now have all my jekyll assets in one place.

One thing I found, though, is that running jekyll clean will blow away the git repo, so it’s not something I want to do on a regular basis, but otherwise, the workflow works well.
